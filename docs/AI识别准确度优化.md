# AI 截图识别：准确度优化与手动修正

## 问题根因

原实现里 `handleImage` 返回的是**前端写死的演示数据**（固定 2 条），根本没有真实模型在跑——所以"3口味×4款式只识别出 2 个"不是模型不准，而是没有模型。

真正的识别管线需要先接上一个多模态大模型，再做工程优化。

---

## 一、如何提升识别准确度

### 1. 选对模型（中文购物截图场景）

| 模型 | 中文 OCR | 结构化输出 | 国内可达性 | 建议 |
|------|---------|-----------|-----------|------|
| 通义千问 Qwen-VL-Max | 强 | 支持 JSON | 直连 | **首选**，中文商品卡片识别效果最好 |
| 智谱 GLM-4V | 强 | 支持 | 直连 | 备选 |
| GPT-4o | 中上 | 支持 `response_format` | 需代理 | 综合能力强 |
| Gemini 1.5 Flash | 中 | 支持 | 需代理 | 便宜、快 |

> 本项目已预置 `api/recognize.ts`（Vercel Serverless），默认对接 Qwen-VL，可一键切换到 GPT-4o。

### 2. 工程手段（比换模型更见效）

按见效程度排序：

**① 强约束 Prompt（已内置 `RECOGNIZE_PROMPT`）**
- 明确要求"识别**所有** SKU，包括每个口味×每个款式的组合，一个都不能漏"
- 要求返回**结构化 JSON 数组**，每个字段带 `confidence` 置信度
- 给出输出示例（few-shot），大幅降低格式错误

**② schema 校验 + 漏识别重试**
- 服务端解析 JSON 失败或条数明显偏少时，带着反馈再问一次：
  "商品区共有约 N 个卡片，你只返回了 M 个，请补全遗漏的规格"
- `api/recognize.ts` 的 `parseItems` 已做字段归一化与容错（剥 markdown、补默认值）

**③ 图片预处理**
- 把截图放大到长边 ~1500px（太小人脸/价格看不清）
- 提高对比度、锐化；裁剪掉顶部状态栏与底部导航，只留商品列表区
- 前端可在上传前用 Canvas 做这一步

**④ 置信度驱动的人机分工**
- 模型对每条返回 `confidence`，低于阈值（0.75）的条目在确认界面**黄色高亮**，引导用户重点核对
- 这样模型只需把"不确定"标出来，不必追求 100% 准

### 3. 前端如何接入

```bash
# .env.local（本地）或 Vercel 环境变量
VITE_RECOGNIZE_ENDPOINT=/api/recognize
```

服务端（Vercel）配置模型密钥：
```bash
RECOGNIZE_PROVIDER=qwen
DASHSCOPE_API_KEY=sk-xxxx        # 通义千问
# 或 OPENAI_API_KEY=sk-xxxx      # GPT-4o
```

未配置端点时，前端自动用演示数据兜底，流程照样跑得通。

## 二、识别错误后的手动修正

无论模型多准，识别结果都必须可编辑才能闭环。已实现「识别确认」环节：

**流程**：上传/拖拽/粘贴截图 → AI 识别 → 弹出**可编辑确认列表** → 用户核对修正 → 一键导入工作台

**修正操作（都很快）**：
- **改**：点击任意字段直接编辑（名称/价格/含量/单位/件数），改完该条置信度自动置为已确认
- **删**：每行末尾垃圾桶图标，删掉 AI 误识别的行
- **增**：底部"补一个 AI 漏掉的规格"，一键加空行手动补录
- **重置**：右上角一键恢复为 AI 原始识别，撤销所有手改
- **低置信度高亮**：把握不足的条目黄色标记，集中核对

只有点击「确认导入 N 条」后，数据才进入工作台参与比价——从源头杜绝脏数据。

相关文件：
- `src/lib/recognize.ts` — 识别逻辑、Prompt、置信度、API 接入点
- `src/components/RecognizeReview.tsx` — 可编辑确认列表
- `api/recognize.ts` — Serverless 转发（密钥在服务端）
